# Broker Gateway/Core Event Contract

PR 1 adds broker-neutral Python contracts under `domain/broker`. These models define the
message shape shared by the future 32-bit Kiwoom Gateway and the Core process, without adding
transport endpoints, Kiwoom imports, strategy decisions, or order execution paths.

## Design Rules

- Contracts serialize to plain dictionaries that are JSON-compatible.
- Contract timestamps are UTC-aware `datetime` values internally and ISO-8601 strings on the
  wire.
- Message IDs are generated by the model layer when omitted.
- `idempotency_key` is preserved by envelopes and order request/result contracts.
- Broker-facing numeric strings such as `"71,000"` or `"+1.25%"` are normalized at the domain
  boundary.
- Domestic stock `code` values normalize Kiwoom-style `A005930` to `005930` and reject invalid
  non-six-digit codes.
- No model imports PyQt, QAxWidget, Kiwoom OpenAPI+, or any concrete gateway implementation.

## Envelopes

### `GatewayEvent` / `EventEnvelope`

Gateway-to-Core event envelope.

| Field | Required | Notes |
| --- | --- | --- |
| `event_id` | Generated if omitted | Unique event identifier with `evt_` prefix. |
| `event_type` | Yes | Event category, for example `price_tick`, `execution`, `condition_match`, or `heartbeat`. |
| `source` | Yes | Gateway or subsystem identifier. |
| `ts` | Generated if omitted | Gateway event timestamp. |
| `payload` | Yes | Broker-neutral payload dictionary. |
| `command_id` | No | Optional command correlation when an event responds to a command. |
| `idempotency_key` | No | Optional duplicate-prevention/correlation key. |

### `GatewayCommand` / `CommandEnvelope`

Core-to-Gateway command envelope.

| Field | Required | Notes |
| --- | --- | --- |
| `command_id` | Generated if omitted | Unique command identifier with `cmd_` prefix. |
| `command_type` | Yes | Command category, for example `request_tr` or `submit_order`. |
| `source` | Yes | Command producer, usually `core`. |
| `ts` | Generated if omitted | Command creation timestamp. |
| `payload` | Yes | Broker-neutral command payload dictionary. |
| `idempotency_key` | No | Optional duplicate-prevention key. |

## Payload Models

### `BrokerPriceTick`

Real-time market tick payload. Required fields are `code`, `name`, `price`, `change_rate`,
`volume`, `trade_value`, `execution_strength`, `best_bid`, `best_ask`, `spread_ticks`,
`day_high`, `day_low`, and `trade_time`; `ts` is generated if omitted.

Prices and quantities are integer-normalized. `price`, `day_high`, and `day_low` must be
positive. `volume`, `trade_value`, `best_bid`, `best_ask`, and `spread_ticks` must be
non-negative. `day_high` must be greater than or equal to `day_low`, and populated best ask must
not be below best bid.

### `BrokerOrderRequest`

Order-intent payload only. It does not execute an order and is not connected to any gateway API.

Required fields are `code`, `side`, and `quantity`; `client_order_id` and `ts` are generated if
omitted. `side` is `BUY` or `SELL`. `order_type` is `LIMIT` by default and may also be
`MARKET`. `LIMIT` orders require a positive `price`; `MARKET` orders may omit `price`.

Optional fields: `account_id`, `idempotency_key`.

### `BrokerOrderResult`

Broker-neutral order acceptance/rejection result. Required fields are `client_order_id`, `code`,
`accepted`, and `status`. `status` is `ACCEPTED` or `REJECTED`.

Optional fields: `broker_order_id`, `reason`, `command_id`, `idempotency_key`.

### `BrokerExecutionEvent`

Broker-neutral execution/fill payload. Required fields are `execution_id`, `broker_order_id`,
`code`, `side`, `quantity`, `price`, and `executed_at`. Quantity and price must be positive.

Optional fields: `client_order_id`, `account_id`, `remaining_quantity`, `commission`, `tax`.

### `BrokerTrRequest`

Broker-neutral TR request payload. Required fields are `tr_code`, `request_name`, and `params`;
`request_id` and `ts` are generated if omitted. `tr_code` is normalized to uppercase.

Optional field: `continuation_key`.

### `BrokerTrResponse`

Broker-neutral TR response payload. Required fields are `request_id`, `tr_code`, `request_name`,
`success`, and `rows`. `rows` must be a list of JSON-compatible objects.

Optional fields: `message`, `continuation_key`.

### `BrokerConditionEvent`

Condition-search membership event. Required fields are `condition_id`, `condition_name`, `code`,
`name`, and `action`. `action` is `ENTER` or `EXIT`.

Optional fields: `price`, `metadata`.

## Serialization

Every contract exposes:

- `to_dict()`: returns a JSON-compatible dictionary.
- `from_dict(data)`: validates and normalizes a dictionary into the domain model.

Common helpers live in `domain/broker/utils.py`:

- `utc_now()` and `timestamp()` for UTC timestamps.
- `new_message_id()` for generated IDs.
- `normalize_payload()` and `normalize_value()` for JSON-compatible payloads.
- `parse_int()` and `parse_float()` for numeric string normalization.

## Explicitly Out of Scope

PR 1 does not implement Gateway/Core HTTP transport, Kiwoom OpenAPI+ integration, PyQt imports,
strategy evaluation, risk decisions, or order execution APIs. Those remain future PR work.
